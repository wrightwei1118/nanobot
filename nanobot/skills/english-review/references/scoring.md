# Scoring System

## Dimensions

Each training session ends with 3 scores (each /10):

- **Fluency** — flow, pauses, natural transitions
- **Grammar** — accuracy, sentence structure, tense/agreement
- **Technical Clarity** — precision, structure, sounds like a real Backend engineer

## Anchor Points

### Fluency

| Score | Description |
|-------|-------------|
| 6/10 | Can convey basic meaning but pauses often, transitions are rough, frequently reorganizes mid-sentence |
| 8/10 | Mostly smooth, occasional hesitation, can naturally complete a technical explanation |
| 9/10 | Fluent expression, natural transitions, rarely needs to stop and restructure |

### Grammar

| Score | Description |
|-------|-------------|
| 6/10 | Noticeable grammar errors but does not seriously impair understanding |
| 8/10 | Few grammar issues, most sentences are structurally stable and clear |
| 9/10 | Grammar is consistently stable, errors are rare and do not affect professional expression |

### Technical Clarity

| Score | Description |
|-------|-------------|
| 6/10 | General idea is understandable but explanation lacks clarity, terminology is vague |
| 8/10 | Can clearly explain main concepts, implementation, or trade-offs with organized technical expression |
| 9/10 | Clear, well-layered explanation — sounds like a real Backend engineer discussing design and trade-offs |

## Scoring Rules

- Use intermediate scores (7, 8.5, etc.) when performance falls between anchors, but stay logically consistent with the anchor descriptions.
- Scores must be evidence-based — cite specific moments from the session.
- Include with every score summary:
  - **Today's strongest point**
  - **Top 1 issue to fix next**
  - **Topic status**: finished / unfinished
